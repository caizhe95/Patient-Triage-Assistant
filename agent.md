# Agent 开发规范

本文件用于约束后续在本项目中工作的 Agent 或协作者。内容只描述长期开发规范，不替代 README、需求文档或项目设计说明。

## Agent 工作原则

- 先阅读相关代码、配置和文档，再开始修改。
- 优先理解现有实现，沿用项目已有风格和技术栈。
- 保持小步修改，避免一次性大范围重写。
- 只修改与任务直接相关的文件。
- 不做无关重构、格式化或目录调整。
- 遇到不确定的业务规则，先说明假设，再谨慎实现。
- 不随意删除数据、索引、报告、本地配置或已有文档。
- 完成后必须说明改了什么、如何验证、还有什么风险。

## 项目约定

- 实际项目根目录为当前包含 `app/`、`data/`、`frontend/`、`scripts/` 的目录。
- 后端入口在 `app/main.py`。
- API 路由在 `app/api/`。
- 核心流程在 `app/graph/`。
- 检索相关代码在 `app/retrieval/`。
- 数据处理和索引构建在 `app/ingestion/`。
- 前端 Demo 在 `frontend/streamlit_app.py`。
- 测试放在 `tests/`。
- 项目现状说明保留在 `PROJECT_STATUS.md`，不要随意删除。

## 开发流程

1. 明确本次任务目标和影响范围。
2. 读取相关入口、调用链和测试。
3. 制定小范围修改方案。
4. 修改代码或文档。
5. 运行可用的测试或手动验证命令。
6. 总结修改内容、验证结果和未覆盖风险。

如果任务涉及 API 返回结构、数据格式、索引构建或前端展示，必须同步检查调用方和测试。

## 测试与运行命令

常用测试命令：

```bash
python -m pytest -q
```

构建索引：

```bash
python scripts/ingest.py
```

启动后端：

```bash
python scripts/run_api.py
```

启动前端：

```bash
python scripts/run_streamlit.py
```

如果当前环境中 `python`、`pytest` 或依赖不可用，不要假装测试通过；需要在最终说明中明确写出未能验证的原因。

## 代码风格

- Python 代码优先使用清晰的类型标注。
- 函数职责保持单一，避免继续扩大“大函数”。
- 命名应直接表达业务含义，避免过度抽象。
- 注释只解释不明显的设计取舍，不复述代码。
- API schema 变更必须同步更新调用方、前端和测试。
- 不引入不必要的新依赖。
- 不为了展示技术而增加复杂度。
- 保持错误处理可读、保守、可调试。

## 数据与配置

- `.env` 是本地配置，不应提交到版本控制。
- `.env.example` 只放示例值，不放真实密钥。
- 不提交 `__pycache__`、IDE 配置、临时文件和无关生成产物。
- 不随意删除 `data/`、`indexes/`、`reports/` 中的内容。
- 修改数据结构或索引流程时，需要说明是否需要重新运行 `python scripts/ingest.py`。
- 涉及外部服务、模型或 API Key 的功能，必须保留可解释的 fallback 或错误提示。

## 审查要求

提交或交付前重点检查：

- 功能是否仍能运行。
- 是否破坏现有 API 返回结构。
- 是否影响 Streamlit 前端展示。
- 是否有测试或手动验证记录。
- 是否引入真实密钥、隐私信息或本地绝对依赖。
- 是否删除了不该删除的数据、索引或报告。
- 是否存在过度设计或无关改动。
- 是否保持代码足够简洁，适合 Demo 项目维护。

## 提交前检查清单

- [ ] 已阅读相关代码和调用链。
- [ ] 改动范围与任务目标一致。
- [ ] 未提交 `.env`、缓存、IDE 配置或无关生成产物。
- [ ] API/schema 变更已同步调用方和测试。
- [ ] 已运行可用测试，或已说明无法运行的原因。
- [ ] 已记录需要重建索引或重启服务的事项。
- [ ] 最终说明包含修改内容、验证结果和剩余风险。
